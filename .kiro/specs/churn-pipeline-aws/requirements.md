# Requirements Document

## Introduction

This document specifies requirements for a standalone, productized churn prediction pipeline on AWS. The pipeline takes any company's raw customer data (a CSV file) and a YAML mapping configuration, then automatically cleans the data, trains a model to predict which customers will leave, scores all customers, explains WHY each prediction was made in plain English, and monitors itself for problems.

The only inputs are: a raw CSV data file + a YAML mapping config.
The output is: a CSV telling the client which customers are likely to leave, why, and what risk level they are.

## Glossary

- **Pipeline**: An automated sequence of steps that takes raw data in and produces predictions out, without manual intervention. Like a factory assembly line where each station does one job and passes the result to the next.
- **Data_Contract**: A standardized definition of what columns the pipeline expects. Think of it as a form with required fields and optional fields — if you don't fill in the required ones, the form gets rejected.
- **Mapping_Config**: A per-client YAML file that translates their column names into the pipeline's standard names. Every company calls things differently — "MonthlyCharges" vs "mrr" vs "monthly_fee" — this file is the Rosetta Stone.
- **Feature_Engineering**: The process of turning raw data into numbers a model can learn from. Raw data says "contract = month-to-month" — the model needs that as a number (e.g., 0, 1, 2). Feature engineering does that translation for every column.
- **XGBoost**: A machine learning algorithm that works like a team of decision-makers. Each one looks at the data and makes a simple yes/no guess about whether a customer will leave. Then the next decision-maker learns from the mistakes of the previous ones. Hundreds of these simple guesses, stacked together, become remarkably accurate. XGBoost is available as a built-in SageMaker algorithm, meaning AWS maintains the code — we just configure and run it.
- **Batch_Transform**: A SageMaker feature that spins up a computer only when needed to score data, then shuts it down. Like hiring a temp worker for one day instead of a full-time employee — cheapest option for periodic predictions.
- **SHAP**: When a model says "Customer X has 85% chance of leaving," the client asks: WHY? SHAP answers by computing how much each piece of information pushed the prediction up or down. Like a jury verdict: "month-to-month contract added +23% risk, only 2 months as customer added +15%, but having a partner reduced risk by -8%." Each feature gets a contribution score.
- **SageMaker_Pipelines**: AWS service that acts as the factory floor manager — it tells each processing step what to do, in what order, and what to do if something fails.
- **Data_Drift**: When incoming data starts looking different from what the model was trained on. Imagine you taught someone to predict ice cream sales using summer data, then asked them to predict in winter — the world changed, so the predictions become unreliable.
- **PSI (Population Stability Index)**: A number measuring how different today's data looks compared to training data. Below 0.1 = same world (safe). 0.1 to 0.2 = something's shifting (watch closely). Above 0.2 = different world (model may be unreliable, consider retraining).
- **MLflow**: An open-source lab notebook for ML. It records every experiment: what settings were used, what score the model got, which version of the model is in production. Without it, you lose track of what worked and what didn't.
- **Risk_Tier**: A human-friendly label (high/medium/low) derived from the raw churn probability. Clients don't want to interpret "0.73" — they want to know "this customer is HIGH risk."
- **AUC-ROC**: Imagine you have a list of all customers, sorted by how likely the model thinks they'll leave. A perfect model would put all the actual leavers at the top. AUC measures how close to perfect your model's sorting is — 1.0 means perfect sorting, 0.5 means random (useless). We require at least 0.70, meaning the model must sort customers significantly better than random chance.
- **F1_Score**: There's a trade-off: you can catch more leavers (but wrongly flag some loyal customers), or be more careful (but miss some leavers). F1 is a single number that balances these two mistakes — it's the harmonic mean of "what fraction of my flags were correct" (precision) and "what fraction of actual leavers did I catch" (recall).
- **Amazon_Bedrock**: AWS's managed service for accessing large language models (like Claude) via API. No infrastructure to manage — you send text in, get text back, pay per request.
- **LLM_Narrative**: A paragraph of plain English explanation generated by a language model (Claude via Bedrock). Instead of "contract_type=month-to-month (+0.23)", the client reads: "This customer has no long-term commitment. Month-to-month customers are 3x more likely to leave than those on annual contracts."
- **Auto_Mapping**: An LLM-powered step where Claude reads the column names and sample rows from a client's CSV and drafts a mapping YAML automatically. A human reviews and approves it before the pipeline uses it.
- **Hyperparameter_Tuning**: The model has settings (like oven temperature for a recipe). Tuning means trying many combinations automatically and picking whichever settings produce the best-scoring model. The system does this itself — no human trial and error needed.
- **Stratified_Split**: When splitting data into training/testing groups, you need both groups to have the same proportion of churned customers. If 20% of customers churned overall, both groups should have ~20% churners. Otherwise the model trains on a skewed view of reality.
- **Scale_Pos_Weight**: When very few customers actually churn (say 15%), the model can get lazy and just predict "won't churn" for everyone — it'd be right 85% of the time! Scale_pos_weight tells the model: "Pay extra attention to the rare churners. Their mistakes cost more." It's a multiplier that makes the model work harder on the minority class.

## Requirements

### Requirement 1: LLM-Powered Auto-Mapping for Client Onboarding

**User Story:** As a pipeline operator onboarding a new client, I want an LLM to automatically draft a mapping YAML by reading the client's column names and sample data, so that I don't have to manually figure out what each column means.

Every company names their columns differently. One calls it "MonthlyCharges", another calls it "mrr", a third calls it "amt_per_month". A human could look at these and immediately know they all mean "how much the customer pays each month." LLMs are good at exactly this kind of pattern recognition — reading column names and sample values, then guessing what standard field each one maps to. This step uses Amazon Bedrock (Claude) to do that guessing, then a human reviews the guess before the pipeline trusts it.

#### Acceptance Criteria

1. WHEN a client uploads a new CSV file to their S3 prefix, THE Auto_Mapping module SHALL read the column names and up to 5 sample rows from the CSV
2. WHEN column names and sample rows are available, THE Auto_Mapping module SHALL send them to Amazon Bedrock (Claude) with a prompt requesting a mapping to the Data_Contract standard fields
3. WHEN the LLM returns a draft mapping, THE Auto_Mapping module SHALL write it to the client's config path as a draft YAML file (suffixed with `.draft.yaml`)
4. THE Auto_Mapping module SHALL require explicit human approval (renaming `.draft.yaml` to `mapping.yaml`) before the pipeline uses the mapping for any data processing
5. IF the LLM call fails (timeout, API error, or unintelligible response), THEN THE Auto_Mapping module SHALL log the failure and proceed without blocking — the operator creates the mapping YAML manually
6. THE Auto_Mapping module SHALL include a confidence score (high/medium/low) for each column mapping in the draft YAML, indicating how certain the LLM is about each mapping

### Requirement 2: Data Contract and Schema Validation

**User Story:** As a pipeline operator, I want a standardized data contract with tiered importance levels, so that I can onboard any client's data regardless of their column naming conventions.

Think of it like a universal plug adapter. Every country (client) has different shaped plugs (column names), but they all need to fit into the same socket (our pipeline). The data contract defines what shape that socket is. Some pins are essential (the pipeline refuses to work without them), others are helpful but optional.

#### Acceptance Criteria

1. THE Data_Contract SHALL define three tiers: Tier 1 (required fields: customer_id, tenure_months, monthly_charges, total_charges, churn_label), Tier 2 (engagement signals: contract_type, payment_method, support_tickets), and Tier 3 (demographics: gender, age_bucket, partner_status)
2. WHEN a client dataset is submitted, THE Pipeline SHALL validate it against the Data_Contract and report which tier fields are present versus missing
3. IF a Tier 1 field is missing from the client dataset, THEN THE Pipeline SHALL reject the dataset with a descriptive error message identifying the missing fields
4. WHEN Tier 2 or Tier 3 fields are missing, THE Pipeline SHALL proceed with available data and log which optional fields were absent
5. WHEN a Mapping_Config YAML file is provided for a client, THE Pipeline SHALL use it to rename the client's raw columns to standard schema field names before validation
6. THE Mapping_Config SHALL support column renaming, type coercion (e.g., converting the string "1234.56" into the number 1234.56), and value mapping (e.g., converting "Yes"/"No" into 1/0)

### Requirement 3: Feature Engineering

**User Story:** As a data scientist, I want standardized feature transformations applied consistently across all clients, so that model performance is reproducible regardless of who runs the pipeline.

Raw data is like raw ingredients — you need to chop, measure, and prep them before cooking. A model can't understand "contract = month-to-month" — it only understands numbers. Feature engineering is that prep work: encoding categories as numbers, scaling values so nothing dominates unfairly, filling in blanks with reasonable guesses, and creating combination features (like monthly_charges × tenure_months, which captures "how much total has this customer paid?").

#### Acceptance Criteria

1. WHEN schema-conformant data is received, THE Feature_Engineering module SHALL encode categorical variables as numbers, scale numerical features to comparable ranges, and create interaction features (e.g., monthly_charges × tenure_months)
2. THE Feature_Engineering module SHALL handle missing values by filling numerical columns with the median (middle value) and categorical columns with the mode (most common value)
3. THE Feature_Engineering module SHALL produce a feature matrix where all columns are numeric and contain no null values
4. WHEN a feature transformation is applied, THE Feature_Engineering module SHALL log the transformation parameters (e.g., what scaling factors were used, what encoding mappings were applied) for reproducibility
5. THE Feature_Engineering module SHALL persist transformation artifacts (the fitted encoders and scalers) to S3 so they can be reloaded during scoring without recomputation

### Requirement 4: Model Training with Hyperparameter Tuning

**User Story:** As a data scientist, I want automated model training with hyperparameter tuning on SageMaker, so that I get the best-performing model without manual experimentation.

Training is teaching the model to recognize patterns — it looks at thousands of customers who already churned or stayed, and learns what features (contract type, tenure, charges) predict churn. Hyperparameter tuning is like trying different oven temperatures and cook times to find the perfect settings: the system automatically trains dozens of models with different settings and picks the one that scores best on a validation set it hasn't seen during training.

XGBoost works by building a team of simple decision trees. Each tree looks at the mistakes of the previous trees and tries to correct them. After hundreds of trees, the team's combined judgment is remarkably accurate. The "gradient boosting" part means each new tree specifically focuses on the examples the previous team got wrong.

#### Acceptance Criteria

1. WHEN the Feature_Engineering step completes, THE Training module SHALL launch an XGBoost training job on SageMaker using the built-in algorithm container
2. THE Training module SHALL perform hyperparameter tuning over at minimum: max_depth (how complex each tree can be), eta (how much each new tree corrects the previous ones — too fast and it overreacts, too slow and it needs thousands of trees), subsample (what fraction of data each tree sees — using less prevents memorization), colsample_bytree (what fraction of features each tree considers), and min_child_weight (how many examples a tree needs before making a decision — prevents it from making rules based on a single customer)
3. WHEN training completes, THE Training module SHALL log all metrics (AUC-ROC, F1, precision, recall) and hyperparameters to MLflow
4. THE Training module SHALL use stratified train/validation/test splits (70/15/15) to preserve class distribution across splits — meaning if 20% of customers churned overall, each split should have approximately 20% churners
5. WHEN a class imbalance is detected (the minority class — usually churners — is below 30% of the dataset), THE Training module SHALL apply the scale_pos_weight parameter proportional to the imbalance ratio, telling XGBoost to pay extra attention to the rare class

### Requirement 5: Model Evaluation and Registry

**User Story:** As a pipeline operator, I want automated quality gates that prevent bad models from reaching production, so that clients always receive reliable predictions.

Before a restaurant serves a dish, a chef tastes it. If it's not good enough, it doesn't go to the table. The evaluation gate is that taste test — the model must prove it can sort customers by churn risk significantly better than random chance, or it gets rejected and never reaches the client.

#### Acceptance Criteria

1. WHEN training completes, THE Evaluation module SHALL compute AUC-ROC, F1-score, precision, and recall on the held-out test set (data the model has never seen during training or tuning)
2. IF the AUC-ROC score is below 0.70 (meaning the model can't sort customers much better than a coin flip), THEN THE Evaluation module SHALL fail the pipeline and log a detailed failure report with all metric values
3. WHEN a model passes the evaluation gate, THE Evaluation module SHALL register it in the SageMaker Model Registry with status "PendingApproval"
4. THE Evaluation module SHALL store a model card (JSON) containing: training date, dataset size, feature list, all metrics, hyperparameters, and data contract version used
5. WHEN a new model is registered, THE Evaluation module SHALL compare its metrics to the currently deployed model and log whether performance improved or degraded

### Requirement 6: Batch Scoring and Client Deliverable

**User Story:** As a client, I want to receive a scored customer list with churn probabilities, risk tiers, and plain-English explanations, so that I can take immediate action on customers likely to leave.

This is the final product the client receives. They get a file that says, for every customer: "Here's how likely they are to leave (a number), here's the human-friendly risk level (high/medium/low), here's exactly why the model thinks so (the factors that pushed the prediction up or down), and here's a paragraph explaining it in plain English."

#### Acceptance Criteria

1. WHEN batch scoring is triggered, THE Scoring module SHALL use SageMaker Batch Transform to generate predictions for all customers in the input dataset
2. THE Scoring module SHALL produce an output CSV containing: customer_id, churn_probability (a decimal between 0.0 and 1.0), risk_tier (high/medium/low), top_3_reasons (SHAP-derived), and narrative_explanation (LLM-generated)
3. THE Scoring module SHALL assign risk_tier based on: high (probability >= 0.7), medium (0.4 <= probability < 0.7), low (probability < 0.4)
4. WHEN generating explanations, THE Scoring module SHALL compute SHAP values for each prediction and extract the top 3 features by absolute contribution as top_3_reasons
5. THE Scoring module SHALL write the output CSV to the client's S3 prefix at path: s3://{bucket}/{client_id}/outputs/scores/{run_date}/predictions.csv
6. IF Batch Transform fails, THEN THE Scoring module SHALL retry once and, if still failing, send an alert notification and log the error details

### Requirement 7: LLM-Powered Narrative Explanations

**User Story:** As a client's business user (non-technical), I want plain-English explanations of why each high-risk customer is predicted to churn, so that I can understand and act on predictions without needing a data scientist.

SHAP gives us numbers like "contract_type=month-to-month (+0.23)". That's useful for a data scientist, but a business user needs to read: "This customer has no long-term commitment. Month-to-month customers are 3x more likely to leave than those on annual contracts. Combined with their low tenure (only 2 months), this customer hasn't built loyalty yet." The LLM takes the raw SHAP numbers and writes that kind of paragraph.

#### Acceptance Criteria

1. WHEN SHAP explanations are computed for scored customers, THE Narrative module SHALL identify the top N highest-risk customers (configurable, default N=50) for narrative generation
2. WHEN generating narratives, THE Narrative module SHALL batch multiple customers into a single Bedrock (Claude) prompt to minimize API calls and cost
3. THE Narrative module SHALL send each batch with: the customer's churn probability, risk tier, top SHAP features with contribution scores, and the feature definitions from the Data_Contract
4. WHEN the LLM returns narrative text, THE Narrative module SHALL validate that each response contains a narrative for every customer in the batch and attach it to the correct customer_id in the output CSV
5. IF the Bedrock API call fails (timeout, rate limit, or malformed response), THEN THE Narrative module SHALL log the failure, leave the narrative_explanation field as "N/A - raw SHAP values available in top_3_reasons", and continue pipeline execution without blocking
6. THE Narrative module SHALL include a system prompt instructing the LLM to explain in non-technical language, reference specific feature values, and keep each narrative under 150 words

### Requirement 8: Monitoring and Drift Detection

**User Story:** As a pipeline operator, I want automated monitoring that alerts me when the world changes (data patterns shift), so that I can retrain before clients receive inaccurate predictions.

When we trained the model, customers looked a certain way — maybe average tenure was 24 months and most were on annual contracts. If next month's batch has average tenure of 6 months and everyone is month-to-month, the model is seeing a world it wasn't trained for. PSI is the smoke detector that catches this before predictions become unreliable.

#### Acceptance Criteria

1. WHEN new scoring data is processed, THE Monitoring module SHALL compute PSI for each feature comparing the current batch against the training distribution
2. IF any feature's PSI exceeds 0.2 (meaning that feature's distribution looks significantly different from training time), THEN THE Monitoring module SHALL flag the feature as drifted and send an alert notification
3. THE Monitoring module SHALL log all drift metrics to MLflow for historical tracking and trend analysis
4. WHEN ground-truth labels become available (we find out who actually churned), THE Monitoring module SHALL compute actual model performance and compare against the evaluation-time metrics
5. IF actual AUC-ROC drops below 0.65 (5 points below the minimum training gate — meaning the model has degraded from "good enough" to "unreliable"), THEN THE Monitoring module SHALL trigger a retraining alert

### Requirement 9: Pipeline Orchestration

**User Story:** As a pipeline operator, I want the end-to-end pipeline orchestrated by SageMaker Pipelines with proper error handling and monitoring, so that all steps execute in the correct order and I'm alerted when something fails.

Think of an assembly line in a factory. Each station does one thing, passes the result to the next station, and if something breaks, the line stops and alerts the supervisor. SageMaker Pipelines is that assembly line manager — it knows the order, handles the handoffs, and raises alarms.

#### Acceptance Criteria

1. THE Pipeline SHALL orchestrate steps in this order: auto_mapping (if new client) → validate_data → feature_engineering → train_model → evaluate_model → register_model → batch_score → compute_shap → generate_narratives → monitor_drift
2. WHEN any step fails, THE Pipeline SHALL halt execution, log the error with step name and error details, and send an alert notification via SNS
3. THE Pipeline SHALL support per-client parameterization via a client_id parameter that determines S3 prefixes, Mapping_Config file location, and all input/output paths
4. WHEN the pipeline completes successfully, THE Pipeline SHALL log total execution time, per-step durations, and final output location to MLflow
5. THE Pipeline SHALL be triggerable via: scheduled cron (weekly), manual API call, or S3 event (new data upload to client prefix)

### Requirement 10: Multi-Client Data Isolation

**User Story:** As a pipeline operator managing multiple clients in a single AWS account, I want strict data isolation between clients, so that no client's data is ever visible to or mixed with another client's data.

Imagine a shared office building where each company has their own locked floor. Even though they're in the same building (AWS account), no one can access another company's floor (data). We use S3 prefixes like separate filing cabinets with different keys — Client A's pipeline execution can only touch Client A's data.

#### Acceptance Criteria

1. THE Pipeline SHALL store all client data under isolated S3 prefixes following the pattern: s3://{bucket}/{client_id}/
2. THE Pipeline SHALL enforce that IAM policies restrict each pipeline execution to only access the S3 prefix matching its client_id parameter
3. WHEN processing data for a client, THE Pipeline SHALL validate that all input and output paths contain the correct client_id prefix before reading or writing
4. THE Pipeline SHALL never co-mingle data from different clients in the same training job, scoring job, or output file

## Out of Scope

- Real-time (sub-second) prediction serving — this version uses batch scoring only
- Custom Docker containers for model training — using SageMaker built-in XGBoost
- Multi-account AWS architecture — single account with per-client S3 prefixes
- Model A/B testing — single model per client for V1
- Web UI dashboard — outputs are CSV files delivered to S3
- Per-customer LLM calls — narratives are batched for cost efficiency
- CatBoost or other algorithms — XGBoost only for V1

## Success Criteria

1. Pipeline successfully processes all three test datasets (IBM Telco 7K rows, e-commerce 37K rows, banking 10K rows) end-to-end without manual intervention
2. Model achieves AUC-ROC >= 0.70 on IBM Telco dataset
3. SHAP explanations generated for all predictions with human-readable feature names
4. LLM narratives generated for top 50 highest-risk customers in plain English
5. Auto-mapping correctly drafts a YAML for at least 2 of the 3 test datasets with >80% of columns correctly mapped
6. New client onboarding requires only: (a) uploading data to their S3 prefix — LLM drafts the mapping, (b) human reviews and approves the mapping YAML
7. Drift detection correctly identifies synthetic drift injected into test data
